### 2.3 消息加密服务 ✅

**文件**：`MessageEncryptionService.java`

**加密流程（发送方）**：
```
1. 生成 AES 会话密钥
2. 使用 AES 加密消息内容
3. 使用接收方公钥加密 AES 密钥
4. 使用发送方私钥对消息签名
5. 计算消息哈希
```

**解密流程（接收方）**：
```
1. 验证发送方公钥指纹
2. 使用接收方私钥解密 AES 密钥
3. 使用 AES 密钥解密消息内容
4. 验证消息哈希（完整性）
5. 验证数字签名（身份）
```

**功能**：
- ✅ 消息加密（端到端）
- ✅ 消息解密
- ✅ 消息完整性验证
- ✅ 消息签名验证
- ✅ 公钥指纹验证

### 2.4 AI 数据访问令牌服务 ✅

**文件**：`AIDataAccessTokenService.java`

**令牌结构**：
```json
{
  "user_id": 10001,
  "session_id": "ai_session_123",
  "device_id": "device_456",
  "data_type": "logistics",
  "allowed_fields": ["status", "location"],
  "timestamp": 1712203815,
  "nonce": "abc123...",
  "expire_time": 1712203820
}
```

**安全机制**：
- ✅ AES 加密令牌内容
- ✅ HMAC-SHA256 签名防篡改
- ✅ Redis 存储（5 秒过期）
- ✅ 一次性使用（SETNX 原子标记）
- ✅ 防重放攻击（nonce + 时间戳）
- ✅ 设备绑定验证

**令牌使用流程**：
```
1. AI 收到问题 → 生成令牌（5 秒有效）
2. 令牌存入 Redis（自动过期）
3. AI 携带令牌访问数据
4. 网关验证令牌：
   - 解密令牌
   - 验证签名
   - 检查是否过期
   - 检查是否已使用（SETNX）
   - 验证设备 ID
5. 验证通过 → 返回数据
6. 令牌标记为已使用
```

   - ⚠️ AI 无法解密端到端加密消息
   - ✅ 这是设计如此（保护隐私）
   - 🔄 AI 助手需要特殊处理（见下方）

---

### 4.2 AI 助手的特殊处理

**问题**：端到端加密后，AI 无法解密用户消息

**解决方案**：
1. **方案 A**：AI 消息不加密（仅 TLS）
   - 优点：AI 可以正常处理
   - 缺点：服务器可见明文

2. **方案 B**：AI 消息使用特殊密钥
   - 用户 + AI 共享密钥
   - AI 可以解密
   - 服务器仍无法解密

3. **推荐方案**：混合方案
   - 用户间聊天：端到端加密
   - 用户-AI 聊天：TLS 加密（服务器可见）
   - AI 数据访问：令牌控制 + 审计日志
