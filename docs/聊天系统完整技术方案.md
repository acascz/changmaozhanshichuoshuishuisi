## 1. 系统概述

### 1.1 系统定位

本聊天系统是一个**电商平台的智能聊天系统**，集成以下核心功能：
- 用户间私信聊天
- 商家客服系统
- 多人团聊/群聊
- AI 智能问答助手
- 电商消息卡片（商品、优惠券、团购）

### 1.2 核心特性

✅ **端到端加密**：只有收发双方能解密消息内容，服务器无法查看明文  
✅ **前后验证机制**：发送前验证接收方身份，接收后验证发送方身份  
✅ **AI 数据隔离**：AI 只能访问当前用户数据，严格用户隔离  
✅ **消息类型丰富**：支持文字、图片、语音、视频、电商卡片、表情  
✅ **实时推送**：WebSocket 实时通信，支持离线消息  
✅ **高并发支持**：支持百万级日活用户，高并发消息处理  

### 1.3 安全原则

1. **最小权限原则**：AI 只能访问回答当前问题所必需的数据
2. **数据隔离原则**：AI 只能访问当前对话用户的数据，绝对不能跨用户访问
3. **数据分级原则**：L1 公开数据、L2 一般敏感数据、L3 高度敏感数据
4. **审计追溯原则**：所有数据访问记录日志，可追溯、可审计
5. **端到端加密**：聊天内容端到端加密，服务器只存储密文

**AI 可访问数据**：
- L1 公开数据：商品信息、物流状态、订单状态（无需授权）
- L2 一般敏感：订单金额、收货地址（脱敏）、物流单号（脱敏）（默认允许）

**AI 安全机制**：
- 严格用户隔离：只能访问当前用户数据
- 数据访问令牌：一次性使用，5 秒过期
- 双重验证：用户身份 + 访问权限
- 审计日志：所有访问记录详细日志
- 异常检测：高频访问、越权访问自动熔断

## 5. 消息加密方案

### 5.1 加密架构

```
端到端加密架构（E2EE）：

发送方（用户 A）                服务器（中转）              接收方（用户 B）
┌─────────────┐              ┌─────────────┐              ┌─────────────┐
│  明文消息   │              │  加密消息   │              │  明文消息   │
│  "你好啊"   │  → 加密 →   │  (密文)     │  → 转发 →   │  "你好啊"   │
└─────────────┘              └─────────────┘              └─────────────┘
       ↓                           ↓                           ↓
  使用接收方公钥加密            无法解密                    使用私钥解密
  或会话密钥加密                仅存储转发                  获取明文
```

### 5.2 加密流程

#### 5.2.1 私聊消息加密

```
步骤 1：发送方准备发送
  用户 A 输入："你好啊"
  
步骤 2：获取接收方公钥
  - 从服务器获取用户 B 的公钥（PublicKey_B）
  - 验证公钥指纹（防止中间人攻击）
  
步骤 3：生成会话密钥
  - 生成随机 AES 密钥（SessionKey）
  - 使用 SessionKey 加密消息："你好啊" → 密文
  
步骤 4：加密会话密钥
  - 使用 PublicKey_B 加密 SessionKey → EncryptedSessionKey
  
步骤 5：发送数据
  - 发送给用户 B：{EncryptedSessionKey + 密文}
  - 发送到服务器（仅存储，无法解密）
  
步骤 6：接收方解密
  - 用户 B 收到数据
  - 使用自己的私钥（PrivateKey_B）解密 EncryptedSessionKey → SessionKey
  - 使用 SessionKey 解密密文 → "你好啊"
```

#### 5.2.2 加密算法

- **消息内容**：AES-256-GCM（对称加密，快速）
- **会话密钥**：RSA-2048 或 ECC-256（非对称加密，安全）
- **哈希验证**：SHA-256（完整性校验）
- **数字签名**：RSA 签名 或 ECDSA

### 5.3 前后验证机制

#### 5.3.1 发送前验证（确认接收方身份）

```
验证流程：

1. 获取接收方公钥时，同时获取公钥指纹
   - 公钥指纹 = SHA256(公钥) 的前 16 位
   - 例：a3f5b8c1d2e4f678

2. 用户界面显示指纹（首次聊天时）
   "您正在与用户 B 聊天
    公钥指纹：a3f5b8c1d2e4f678
    [确认] [取消]"

3. 若用户之前验证过，自动确认
   - 本地缓存公钥指纹
   - 下次聊天时比对指纹
   - 指纹一致 → 自动发送
   - 指纹不一致 → 告警"对方公钥已变更，可能存在安全风险"
```

#### 5.3.2 接收后验证（确认发送方身份）

```
验证流程：

1. 收到消息时，验证数字签名
   - 发送方使用私钥对消息签名
   - 接收方使用发送方公钥验证签名
   - 签名验证通过 → 确认是发送方本人
   - 签名验证失败 → 告警"消息可能被篡改"

2. 消息完整性验证
   - 发送方计算消息哈希值（SHA-256）
   - 接收方重新计算哈希值并比对
   - 哈希值一致 → 消息完整
   - 哈希值不一致 → 告警"消息已损坏或被篡改"

3. 用户界面提示
   ✓ 验证通过：显示"已验证来自用户 A"
   ✗ 验证失败：显示"⚠️ 消息来源无法验证"
```

### 5.4 密钥管理

#### 5.4.1 密钥生成与存储

```
用户注册时：
1. 客户端生成密钥对（公钥 + 私钥）
   - 算法：RSA-2048 或 ECC-256
   - 公钥：可以公开
   - 私钥：绝对保密

2. 私钥加密存储
   - 使用用户密码派生的密钥加密私钥
   - 加密后的私钥存储到本地
   - 服务器不存储私钥（只存公钥）

3. 公钥上传服务器
   - 服务器存储用户公钥
   - 其他用户可以获取
```

#### 5.4.2 密钥备份与恢复

```
密钥备份（可选）：
1. 用户使用备用设备登录时
2. 通过扫码或验证码方式验证身份
3. 从原设备导出私钥（加密）
4. 导入到新设备

密钥丢失处理：
1. 用户重置密钥对
2. 通知所有联系人（公钥已变更）
3. 历史消息无法解密（提前提示用户备份）
```

### 5.5 群聊加密方案

```
群主创建群组时：
1. 生成群组密钥（GroupKey）
2. 使用每个成员的公钥加密 GroupKey
3. 分发给每个成员（成员用自己的私钥解密获取 GroupKey）

发送群消息：
1. 使用 GroupKey 加密消息
2. 发送到群聊
3. 所有成员用 GroupKey 解密

成员变动：
- 新成员加入：重新生成 GroupKey，用新成员公钥加密分发
- 成员退出：重新生成 GroupKey（防止退出后仍能解密）
```

## 6. AI 数据访问安全方案

### 6.1 核心安全原则

1. **最小权限原则**：AI 只能访问回答当前问题所必需的数据
2. **数据隔离原则**：AI 只能访问当前对话用户的数据，绝对不能跨用户访问
3. **数据分级原则**：L1/L2/L3 三级数据分类，差异化保护
4. **审计追溯原则**：所有数据访问记录日志，可追溯、可审计
5. **用户知情原则**：用户知道 AI 访问了哪些数据，可随时查看和清除

### 6.2 数据分级与脱敏

#### 6.2.1 L1 - 公开数据（无需脱敏）

| 数据类别 | 字段列表 | AI 访问策略 |
|---------|---------|-----------|
| 商品基础信息 | 商品 ID、商品名称、描述、图片、分类 | 直接访问 |
| 价格信息 | 商品价格、原价、折扣 | 直接访问 |
| 订单状态 | 订单状态（待付款/待发货/待收货/已完成） | 直接访问 |
| 物流状态 | 物流状态（已发货/运输中/已签收）、物流公司 | 直接访问 |
| 活动信息 | 活动名称、类型、开始时间、结束时间 | 直接访问 |

#### 6.2.2 L2 - 一般敏感（必须脱敏）

| 数据类别 | 字段列表 | 脱敏规则 | AI 访问策略 |
|---------|---------|---------|-----------|
| 收货地址 | 省、市、区 | 显示到区级，隐藏街道<br>例：`北京市朝阳区***` | 默认允许 |
| 联系电话 | 手机号 | 隐藏中间 4 位<br>例：`138****1234` | 默认允许 |
| 订单号 | 订单编号 | 显示前后各 4 位<br>例：`2026****5678` | 默认允许 |
| 物流单号 | 快递单号 | 显示前后各 4 位<br>例：`SF1234****5678` | 默认允许 |
| 订单金额 | 实付金额 | 隐藏分位<br>例：`¥1,234.*` | 默认允许 |

#### 6.2.3 L3 - 高度敏感（严格保护）

| 数据类别 | 字段列表 | 访问策略 |
|---------|---------|---------|
| 详细地址 | 街道、门牌号、楼栋号 | 默认禁止，特殊情况需授权 + 二次验证 |
| 完整电话 | 完整手机号 | 默认禁止，特殊情况需授权 + 二次验证 |
| 身份证号 | 身份证号码 | 永不返回给 AI |
| 支付账号 | 支付宝、微信、银行卡号 | 永不返回给 AI |
| 密码 | 登录密码、支付密码 | 永不返回给 AI |

### 6.3 用户数据隔离机制

#### 6.3.1 严格的用户边界

```
核心原则：
AI 只能访问当前对话用户的数据，绝对不能访问其他人的任何数据！

技术实现：
1. 会话绑定用户 ID
   - AI 会话创建时，绑定当前用户 ID
   - 会话 ID 与用户 ID 强关联

2. 数据访问强制用户隔离
   - 所有 AI 数据访问，强制添加 WHERE user_id = {当前会话 user_id}
   - 即使 AI 服务忘记添加，数据网关也会强制添加

3. 用户切换自动隔离
   - 用户切换账号 → 旧会话失效 → 新会话绑定新用户 ID
   - AI 无法访问旧账号数据
```

#### 6.3.2 五道防线

```
防线 1：会话绑定
- AI 会话与用户 ID 强绑定
- 切换用户 = 切换会话 = 数据隔离

防线 2：令牌绑定
- 访问令牌与用户 ID 绑定
- 令牌只能访问指定用户的数据

防线 3：网关强制
- 数据访问网关强制添加 user_id 条件
- 即使 AI 忘记添加，网关也会补上

防线 4：数据库视图
- AI 专用视图自动过滤用户数据
- 数据库层面保证只能访问当前用户数据

防线 5：审计验证
- 每次访问记录 session_user_id 和 query_user_id
- 不一致立即告警（高风险）
```

### 6.4 数据访问令牌（DAT）

#### 6.4.1 令牌结构

```json
{
  "token": "加密后的令牌字符串",
  "encrypted_data": {
    "user_id": "加密的用户 ID",
    "session_id": "当前会话 ID",
    "device_id": "设备 ID"
  },
  "plaintext": {
    "timestamp": 1712203815,
    "nonce": "abc123",
    "data_type": "logistics",
    "allowed_fields": ["status", "location"]
  },
  "signature": "HMAC-SHA256 签名"
}
```

#### 6.4.2 令牌使用流程

```
步骤 1：AI 收到问题
  → 从当前请求中提取用户 Token、设备 ID、会话 ID
  → 验证用户登录状态
  
步骤 2：生成访问令牌
  → 系统根据问题类型生成令牌
  → 使用 AES 加密敏感字段
  → 添加 HMAC 签名
  → 令牌存入 Redis（5 秒过期，自动清理）
  
步骤 3：AI 访问数据
  → 携带令牌访问数据接口
  → 数据网关验证：
    - 解密令牌
    - 验证签名
    - 检查是否过期
    - 检查是否已使用（Redis SETNX）
    - 检查设备 ID 是否匹配
    - 检查会话是否有效
  
步骤 4：返回数据
  → 验证通过后，返回脱敏数据
  → 令牌标记为已使用（Redis 中设置 flag）
  → 记录审计日志
  
步骤 5：令牌失效
  → Redis 自动过期（5 秒）
  → 或立即标记为已使用
```

### 6.5 异常检测 + 熔断

#### 6.5.1 异常判定标准（量化）

| 异常类型 | 判定标准 | 风险等级 | 处理措施 |
|---------|---------|---------|---------|
| 高频访问 | 同一 AI 服务 1 分钟访问>20 个不同用户数据 | 中风险 | 告警 + 限流 |
| 高频访问 | 同一 AI 服务 1 分钟访问>50 个不同用户数据 | 高风险 | 熔断 + 人工审计 |
| 敏感数据访问 | 1 分钟内访问 L3 数据>3 次 | 中风险 | 告警 + 二次验证 |
| 敏感数据访问 | 1 分钟内访问 L3 数据>10 次 | 高风险 | 熔断 + 人工审计 |
| 越权访问 | 尝试访问非当前用户数据 | 高风险 | 立即拒绝 + 告警 + 记录 |
| 令牌验证失败 | 1 分钟内失败>5 次 | 中风险 | 临时封禁 5 分钟 |
| 令牌验证失败 | 1 分钟内失败>20 次 | 高风险 | 熔断 + 人工审计 |

#### 6.5.2 熔断降级方案

```
检测到异常 → 立即熔断 → 降级处理 → 告警 → 人工审计

降级处理：
- 对用户：返回"系统繁忙，请稍后再试"
- 对 AI：临时关闭数据访问权限，只能回答通用问题
- 对运维：发送告警通知（短信 + 邮件）

熔断恢复：
- 中风险：30 分钟后自动恢复
- 高风险：必须人工审计后手动恢复
```

## 7. 实时通信方案

### 7.1 WebSocket 配置

```java
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {
    
    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        // WebSocket 端点（支持 SockJS 降级）
        registry.addEndpoint("/ws/chat")
                .setAllowedOrigins("*")
                .withSockJS()
                .setSessionCookieNeeded(true)
                .setHeartbeatTime(25000);
    }
    
    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        // 启用简单消息代理
        registry.enableSimpleBroker("/topic", "/queue", "/user");
        
        // 设置应用前缀
        registry.setApplicationDestinationPrefixes("/app");
        
        // 设置用户目的地前缀
        registry.setUserDestinationPrefix("/user");
    }
}
```

### 7.2 消息推送流程

```
用户 A 发送消息给用户 B：

1. 用户 A 通过 WebSocket 发送消息
   STOMP 消息：SEND
   Destination: /app/chat.sendMessage
   Body: {sessionId, content, type}

2. 服务器接收消息
   - 验证用户身份
   - 加密消息内容
   - 存储到数据库

3. 服务器推送给用户 B
   STOMP 消息：MESSAGE
   Destination: /user/queue/messages
   Body: {messageId, sessionId, encryptedContent, type}

4. 用户 B 收到消息
   - 解密消息内容
   - 验证发送方身份
   - 显示到聊天界面
```

### 7.3 离线消息处理

```
用户 B 离线时：

1. 消息加密后存储到服务器
2. 服务器记录用户 B 的未读消息数
3. 用户 B 上线后：
   - 连接 WebSocket
   - 拉取未读消息列表
   - 逐条解密并显示
   - 更新已读状态

离线消息保存期限：
- 默认：7 天
- 用户可设置：永久保存（需额外加密备份）
- 7 天后自动删除
```

## 9. 特殊场景处理

### 9.1 消息撤回

```
消息撤回流程：

1. 发送方点击"撤回"（2 分钟内）
2. 发送撤回请求到服务器
3. 服务器：
   - 验证发送方身份
   - 验证是否在撤回时限内
   - 更新消息状态为"撤回"
   - 删除服务器存储的密文
4. 通知接收方："对方撤回了一条消息"
5. 接收方本地删除消息

注意：
- 撤回前，接收方已解密的消息无法强制删除
- 提前提示用户："撤回后，对方本地可能仍保留消息"
```

### 9.2 消息删除

```
消息删除流程：

1. 用户点击"删除"
2. 发送删除请求到服务器
3. 服务器：
   - 更新消息状态为"删除"（仅对当前用户）
   - 不影响其他用户
4. 用户本地删除消息

注意：
- 删除仅对自己可见
- 其他用户仍可查看消息
```

### 9.3 消息转发

```
消息转发流程：

1. 用户选择要转发的消息
2. 选择目标会话
3. 发送转发请求
4. 服务器：
   - 复制消息内容
   - 重新加密（使用新接收方的公钥）
   - 发送到目标会话
5. 目标会话收到转发的消息

注意：
- 转发消息需重新加密
- 转发消息标记"转发"标识
```

### 9.4 多设备登录

```
多设备登录方案：

方案：多设备共享密钥对
- 所有设备使用同一密钥对
- 私钥在所有设备间同步（加密传输）
- 优点：发送方只需加密一次

实现：
1. 主设备生成密钥对
2. 新设备登录时，从主设备同步私钥（加密传输）
3. 所有设备使用同一私钥解密

消息推送：
- 所有在线设备都接收消息
- 每个设备独立解密
- 已读状态同步到所有设备
```

### 9.5 用户未登录

```
用户未登录时：

1. 无法使用聊天功能
2. 提示："请先登录后使用聊天功能"
3. 跳转到登录页面

AI 助手：
- 未登录时可以使用
- 但无法访问用户数据（订单、物流等）
- 只能回答通用问题
```

## 12. 技术挑战与解决方案

### 12.1 挑战 1：端到端加密的性能开销

**问题**：
- 非对称加密（RSA）计算开销大
- 高并发场景下可能影响响应速度

**解决方案**：
- 混合加密：AES + RSA
  - 消息内容使用 AES 加密（快速）
  - AES 密钥使用 RSA 加密（安全）
- 性能优化：
  - AES-256-GCM 硬件加速
  - 批量加密优化
  - 加密结果缓存

---

### 12.2 挑战 2：AI 数据访问安全

**问题**：
- AI 需要访问用户数据
- 但必须严格隔离，防止越权

**解决方案**：
- 五道防线：会话绑定、令牌绑定、网关强制、数据库视图、审计验证
- 数据分级：L1/L2/L3，差异化保护
- 异常检测：高频访问、越权访问自动熔断

---

### 12.3 挑战 3：高并发消息处理

**问题**：
- 百万级日活用户
- 高并发消息收发

**解决方案**：
- 消息队列：RabbitMQ / Kafka
- 分库分表：按用户 ID 分片
- 缓存优化：Redis 缓存热点数据
- 消息合并：批量推送消息

---

### 12.4 挑战 4：多设备同步

**问题**：
- 用户多设备登录
- 消息同步、已读状态同步

**解决方案**：
- 多设备共享密钥对
- 消息推送所有设备
- 已读状态实时同步
- 冲突处理：最后操作优先

---

### 12.5 挑战 5：离线消息

**问题**：
- 用户离线时消息存储
- 上线后消息拉取

**解决方案**：
- 离线消息存储：7 天
- 消息索引：按用户 ID + 时间排序
- 增量拉取：只拉取未读消息
- 消息分页：每次拉取 50 条

### 附录 A：加密算法选型

| 用途 | 算法 | 密钥长度 | 说明 |
|-----|------|---------|------|
| 消息加密 | AES-256-GCM | 256 位 | 对称加密，快速 |
| 密钥加密 | RSA / ECC | 2048 位 / 256 位 | 非对称加密，安全 |

| 数字签名 | RSA / ECDSA | 2048 位 / 256 位 | 身份验证 |
