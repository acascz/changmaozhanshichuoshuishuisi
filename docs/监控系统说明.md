# 监控系统：操作日志 + 全链路追踪

## 架构设计

### 核心组件

#### 1. OperationLog（实体类）
操作日志数据模型，包含：
- 用户信息（userId, username）
- 操作信息（operationType, module, description）
- 请求信息（method, url, params, ipAddress, userAgent）
- 响应信息（result, executionTime, status, errorMessage）
- 追踪信息（traceId）

#### 2. TraceUtil（追踪工具类）
全链路追踪核心：
```java
// 生成追踪 ID
String traceId = traceUtil.generateTraceId();
// 格式：TRACE-时间戳 - 随机数
// 示例：TRACE-1712131200000-A1B2C3D4

// 设置追踪 ID
traceUtil.setTraceId(traceId);

// 获取追踪 ID（自动创建）
String currentTraceId = traceUtil.getTraceId();

// 清除追踪 ID
traceUtil.clear();
```

#### 3. OperationLogService（日志服务）
异步记录操作日志：
- `@Async` 异步执行，不影响主业务
- 支持记录成功/失败日志
- 自动关联追踪 ID

#### 4. OperationLogInterceptor（日志拦截器）
自动拦截所有请求：
- preHandle：生成追踪 ID，记录开始时间
- afterCompletion：记录完整日志，清理资源

## 工作流程

### 完整请求链路
```
1. 用户发起请求
   ↓
2. OperationLogInterceptor.preHandle()
   ├─ 生成追踪 ID（TRACE-xxx）
   ├─ 记录请求开始时间
   └─ 设置 MDC（日志追踪）
   ↓
3. 业务处理（Controller → Service → Mapper）
   ↓
4. OperationLogInterceptor.afterCompletion()
   ├─ 计算执行时长
   ├─ 获取用户信息
   ├─ 获取请求参数
   ├─ 记录操作日志（异步）
   └─ 清理追踪 ID
   ↓
5. OperationLogService.logOperation()（异步线程）
   ├─ 设置追踪 ID
   ├─ 插入数据库
   └─ 记录调试日志
```

### 日志记录示例

#### 成功请求
```java
operationLogService.logSuccess(
    userId,           // 用户 ID
    username,         // 用户名
    "POST",           // 操作类型
    "订单",           // 模块
    "/api/order/create", // 描述
    "POST",           // 方法
    "/api/order/create", // URL
    "{\"items\":[...]}", // 参数
    "success",        // 结果
    150L,             // 执行时长 (ms)
    "192.168.1.100",  // IP 地址
    "Mozilla/5.0..."  // User-Agent
);
```

#### 失败请求
```java
operationLogService.logFailure(
    userId,
    username,
    "POST",
    "支付",
    "/api/payment/callback",
    "POST",
    "/api/payment/callback",
    "{\"orderNo\":\"...\"}",
    "签名验证失败",  // 错误信息
    "192.168.1.100",
    "Mozilla/5.0..."

```

## 数据库表结构

### operation_log 表
```sql
CREATE TABLE `operation_log` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `user_id` BIGINT,           -- 用户 ID
  `username` VARCHAR(100),    -- 用户名
  `operation_type` VARCHAR(50), -- 操作类型
  `module` VARCHAR(50),       -- 模块
  `description` VARCHAR(500), -- 描述
  `method` VARCHAR(10),       -- 请求方法
  `url` VARCHAR(500),         -- URL
  `params` TEXT,              -- 参数
  `result` VARCHAR(500),      -- 结果
  `execution_time` BIGINT,    -- 执行时长 (ms)
  `ip_address` VARCHAR(50),   -- IP 地址
  `user_agent` VARCHAR(500),  -- User-Agent
  `status` TINYINT,           -- 状态 (0-失败 1-成功)
  `error_message` TEXT,       -- 错误信息
  `trace_id` VARCHAR(100),    -- 追踪 ID
  `create_time` DATETIME      -- 创建时间
  
  INDEX `idx_user_id` (`user_id`),
  INDEX `idx_trace_id` (`trace_id`),
  INDEX `idx_module` (`module`),
  INDEX `idx_create_time` (`create_time`)
);
```

## 使用示例

### 1. 查询用户操作历史
```java
@Autowired
private OperationLogService operationLogService;

// 查询用户最近 100 条操作
List<OperationLog> logs = operationLogService.getUserLogs(userId);
```

### 2. 全链路追踪
```java
// 根据追踪 ID 查看完整链路
List<OperationLog> traceChain = operationLogService.getTraceChain("TRACE-1712131200000-A1B2C3D4");

// 输出：
// TRACE-xxx | 2024-04-03 20:00:00 | POST /api/order/create | 150ms | 成功
```

### 3. 查询模块操作日志
```java
// 查询订单模块的所有操作
List<OperationLog> orderLogs = operationLogService.getModuleLogs("订单");
```

## 性能优化

### 1. 异步记录
- 使用 `@Async` 异步执行
- 不阻塞主业务流程
- 日志记录失败不影响业务

### 2. 定时清理
```java
@Scheduled(cron = "0 0 2 * * ?") // 每天凌晨 2 点
public void cleanOldLogs() {
    int count = operationLogMapper.deleteBeforeDays(30);
    log.info("清理了 {} 条旧日志", count);
}
```
- 自动清理 30 天前的日志
- 防止数据库膨胀

### 3. 索引优化
- `idx_user_id`：快速查询用户日志
- `idx_trace_id`：快速追踪链路
- `idx_module`：快速查询模块日志
- `idx_create_time`：时间范围查询

## 注意事项

### 1. 敏感信息保护
⚠️ 请求参数中的敏感信息（密码、Token）应该脱敏  
⚠️ 不要在日志中明文存储用户隐私

### 2. 性能影响
⚠️ 异步线程池大小要合理配置  
⚠️ 数据库写入压力要考虑  
⚠️ 建议生产环境使用消息队列

### 3. 日志保留
⚠️ 生产环境建议保留 30 天  
⚠️ 重要操作日志建议永久保留  
⚠️ 定期备份和清理

### 4. 分布式追踪
⚠️ 当前实现是单机版  
⚠️ 分布式系统需要使用 Zipkin、SkyWalking 等  
⚠️ 需要跨服务传递 traceId
