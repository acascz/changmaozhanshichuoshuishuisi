# 活动系统：安全接口框架说明

## 架构设计

### 核心目标
本活动系统框架旨在提供：
1. **灵活的价格计算**：支持多种活动类型（满减、折扣、秒杀、团购等）
2. **严格的安全验证**：防止配置错误导致资金损失
3. **清晰的职责分离**：系统提供框架，运维/运营负责业务逻辑

### 计算公式
```
最终价格 = (Σ(商品单价 × 数量) - Σλ) × ρ1 × ρ2 × ...
```

其中：
- **λ (lambda)** = 固定减免金额（如满 100 减 20 中的 20）
- **ρ (rho)** = 折扣系数（如 9 折是 0.9，7 折是 0.7）

## 核心组件

### 1. ActivityConfig（活动配置实体）

```java
@Data
public class ActivityConfig {
    private Long id;                  // 活动 ID
    private String name;              // 活动名称
    private Integer type;             // 活动类型 (1-满减，2-折扣，3-秒杀...)
    private BigDecimal discountRate;  // 折扣系数 ρ (默认 1.0)
    private BigDecimal fixedDiscount; // 固定减免 λ (默认 0)
    private Integer stackable;        // 是否可叠加
    private Integer priority;         // 优先级 (数字越小优先级越高)
    private Integer status;           // 活动状态 (0-未开始，1-进行中...)
    private Date startTime;           // 开始时间
    private Date endTime;             // 结束时间
    private String securitySign;      // 安全签名
    // ... 其他字段
}
```

### 2. ActivityPriceCalculator（价格计算器）

核心方法：
```java
/**
 * 计算活动价格
 * @param originalPrice 商品原价总额
 * @param activities 活动配置列表
 * @return 活动后价格
 */
public BigDecimal calculateActivityPrice(BigDecimal originalPrice, 
                                         List<ActivityConfig> activities)
```

计算流程：
1. 按优先级排序活动
2. 验证每个活动的有效性
3. 应用折扣：`(price - λ) × ρ`
4. 确保价格不为负数

### 3. ActivityController（活动接口）

提供的 API 接口：
- `POST /api/activity/calculate` - 计算活动价格
- `POST /api/activity/validate` - 验证活动配置
- `GET /api/activity/test/calculate` - 测试计算（开发用）

### 接入步骤

#### 步骤 1：理解活动类型

| 类型 | type 值 | 计算方式 | 示例 |
|------|---------|----------|------|
| 满减 | 1 | price - λ | 满 100 减 20 |
| 折扣 | 2 | price × ρ | 9 折 = price × 0.9 |
| 秒杀 | 3 | 固定秒杀价 | 原价 100，秒杀价 50 |
| 团购 | 4 | 根据人数阶梯定价 | 3 人团 8 折，5 人团 7 折 |
| 优惠券 | 5 | price - λ 或 price × ρ | 满 200 减 50 |

#### 步骤 2：配置活动

**方式一：数据库直接配置**
```sql
INSERT INTO activity_config (
    name, type, discount_rate, fixed_discount, 
    start_time, end_time, status, priority
) VALUES (
    '双 11 大促', 
    1,              -- 满减
    1.0000,         -- 无折扣
    50.00,          -- 减 50 元
    '2024-11-11 00:00:00',
    '2024-11-11 23:59:59',
    1,              -- 进行中
    1               -- 优先级高
);
```

**方式二：通过 API 配置**
```java

activity.setName("双 11 大促");
activity.setType(1);
activity.setDiscountRate(new BigDecimal("1.0"));
```

#### 步骤 3：实现自定义业务逻辑

## 安全警告 ⚠️

### 1. 配置安全

**必须验证的内容：**
```java
// 1. 检查折扣率范围
if (discountRate < 0 || discountRate > 10) {
    throw new RuntimeException("折扣率异常：必须在 0-10 之间");
}

// 2. 检查固定减免金额
if (fixedDiscount < 0) {
    throw new RuntimeException("减免金额不能为负数");
}

// 3. 检查活动后价格
if (finalPrice < 0) {
    throw new RuntimeException("活动后价格不能为负数");
}
```

### 2. 常见配置错误案例

#### ❌ 错误案例 1：0 折活动
```sql
-- 错误配置：折扣率为 0，商品免费！
INSERT INTO activity_config (name, discount_rate) 
VALUES ('免费赠送', 0.0000);

-- 正确配置：最低折扣不能低于成本
VALUES ('成本价促销', 0.3000); -- 最低 3 折
```

#### ❌ 错误案例 2：叠加漏洞
```sql
-- 错误配置：多个可叠加活动导致负价格
活动 A: 满 100 减 50, 可叠加
活动 B: 满 100 减 60, 可叠加
-- 用户买 100 元商品：100 - 50 - 60 = -10 元

-- 正确配置：设置不可叠加或限制条件
活动 A: stackable = 0 (不可叠加)
活动 B: stackable = 0 (不可叠加)
```

#### ❌ 错误案例 3：时间配置错误
```sql
-- 错误配置：结束时间早于开始时间
start_time: '2024-12-01 00:00:00'
end_time:   '2024-11-01 00:00:00'  -- 活动永远不会开始！

-- 正确配置
end_time > start_time
```

### 3. 安全签名机制

**防止配置被篡改：**
```java
// 生成签名
String sign = SignUtil.sign(
    activity.getId() + "|" + 
    activity.getDiscountRate() + "|" + 
    activity.getFixedDiscount() + "|" + 
    activity.getPriority()
);
activity.setSecuritySign(sign);

// 验证签名
if (!SignUtil.verify(activity)) {
    throw new RuntimeException("活动配置被篡改，拒绝执行");
}
```

## 常见问题 FAQ

### Q1: 如何设置活动优先级？
**A:** 通过 `priority` 字段控制，数字越小优先级越高：
```sql
-- 优先级最高（最先应用）
UPDATE activity_config SET priority = 1 WHERE name = '双 11 大促';

-- 优先级最低（最后应用）
UPDATE activity_config SET priority = 999 WHERE name = '日常促销';
```

### Q2: 活动可以叠加吗？
**A:** 通过 `stackable` 字段控制：
- `stackable = 0`：不可叠加（默认）
- `stackable = 1`：可与其他活动叠加

### Q3: 如何限制每人参与次数？
**A:** 设置 `user_limit` 字段：
```sql
-- 每人限享 1 次
UPDATE activity_config SET user_limit = 1 WHERE id = 1;

-- 不限次数
UPDATE activity_config SET user_limit = 0 WHERE id = 1;
```

### Q4: 活动配置错误导致损失怎么办？
**A:** 
1. 立即下线活动：`UPDATE activity_config SET status = 3 WHERE id = ?;`
2. 查看操作日志：`SELECT * FROM operation_log WHERE module = '活动';`
3. 回滚到上一个正确版本
4. 分析原因，修复后重新上线
